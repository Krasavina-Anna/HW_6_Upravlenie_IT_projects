## Сравнение подходов к тестированию для реальной платёжной системы

### 1. Сравнительная таблица

| Вид тестов | Инструмент | Что проверяет | Время | Сложность | Целесообразность для FastPay |
|------------|------------|---------------|-------|-----------|------------------------------|
| **Модульные** | pytest | Изолированная логика (Луна, комиссия) | < 1 сек | Низкая | **Обязательны** |
| **Интеграционные** | pytest + requests | Связь с банком, БД, API | 1-5 сек | Средняя | **Обязательны** |
| **Нагрузочные** | locust | Пропускная способность (5000 tps) | 10-60 мин | Высокая | При SLA > 1000 tps |
| **Контрактные** | pact-verifier | Совместимость с API банка | < 10 сек | Средняя | **Обязательны** |
| **E2E** | behave | Полный сценарий оплаты | 30-60 сек | Высокая | Нецелесообразны |

### 2. Подробный анализ

#### Модульные тесты (pytest)
- **Плюсы:** дёшево, быстро, легко писать
- **Минусы:** не проверяют интеграции
- **Для FastPay:** покрывают validate_card() и calculate_fee() — 80% багов

#### Интеграционные тесты
- **Плюсы:** проверяют реальное взаимодействие
- **Минусы:** требуют тестовое окружение (mock-банк)
- **Для FastPay:** без них нельзя убедиться, что банк получил транзакцию

#### Нагрузочные тесты (locust)
- **Плюсы:** выявляют узкие места
- **Минусы:** дорогие, долгие
- **Для FastPay:** нужны только при высокой нагрузке (>1000 tps)

#### Контрактные тесты (pact-verifier)
- **Плюсы:** ловят изменения API до релиза
- **Минусы:** требуют поддержки от банка
- **Для FastPay:** критичны — ошибка в API банка = потеря денег

#### E2E тесты (behave)
- **Плюсы:** проверяют полный сценарий
- **Минусы:** медленные, нельзя откатить деньги
- **Для FastPay:** нецелесообразны — ручное тестирование эффективнее

### 3. Выводы

Для реальной платёжной системы FastPay:

1. **Модульные тесты** — база. Покрывают 80% дефектов при затратах 1% бюджета.

2. **Интеграционные тесты** — обязательны. Платёжный процессор без проверки связи с банком не имеет смысла.

3. **Контрактные тесты** — must-have. API банка меняется независимо, pact выявляет несовместимости до выкатки.

4. **Нагрузочные тесты** — только если есть жёсткое SLA по производительности.

5. **E2E тесты** — антипаттерн для платежей (нельзя автоматически откатить деньги).

**Итог:** Минимальный набор = Unit + Integration + Contract. Остальное — по потребностям.
